# 收智通（SZTong）智能环保回收平台 — 项目介绍文档

## 一、项目背景与概述

### 1.1 项目背景

随着城市化进程的加速和消费水平的提升，城市固体废弃物产生量持续增长，垃圾分类与资源回收已成为影响城市可持续发展的重要议题。传统的垃圾处理方式往往是混合收集、统一填埋或焚烧，这种粗放型的处理模式不仅浪费了大量可回收资源，还会对土壤、水体和大气造成持续性污染。党的二十大报告明确提出“推动绿色发展，促进人与自然和谐共生”的战略目标，全国各地都在积极推进垃圾分类试点和再生资源回收体系建设。

在此背景下，依托人工智能技术和移动互联网平台，构建智能化、便捷化的居民回收服务体系，具有重要的现实意义和社会价值。「收智通」（英文品牌名：SZTong）正是在这一时代需求指引下诞生的智能环保回收平台，旨在通过 AI 技术赋能，让每一位居民都能便捷参与垃圾分类与资源回收，共同为建设美丽中国贡献力量。

### 1.2 项目定位

「收智通」是一款面向 C 端（普通居民用户）的智能环保回收平台 Web 应用，基于当前主流的 Vue 3 前端技术栈构建，结合 Node.js (Express) 轻量级后端代理服务，为用户提供一站式的环保服务体验。平台整合了 AI 智能识别、预约上门回收、公益捐赠、旧物改造等核心功能，致力于打造“让回收触手可及、让环保成为一种生活方式”的用户体验。

从技术实现角度而言，本项目的前端通过本地存储和前端API封装层实现业务功能演示，而 AI 能力（如 DeepSeek 智能问答、MiniMax 图像识别）则通过后端代理服务进行调用。这种前后端分离配合代理转发的开发模式，不仅保护了核心的 API 密钥安全，也有助于快速验证产品思路、探索用户体验方案，同时也为后续完整后端业务逻辑的开发提供了清晰的对接基础。

### 1.3 项目愿景

「收智通」的核心理念是“智慧回收，绿色生活”。我们希望通过技术手段降低回收门槛、提升回收效率、优化用户参与体验，最终实现以下目标：

- **降低参与门槛**：通过 AI 智能识别，帮助用户快速准确地判断垃圾类别，消除“不会分、不敢分”的顾虑
- **提升便利性**：提供预约上门回收服务，让用户足不出户即可完成回收
- **增强参与感**：通过环保积分、成就系统、社区互动等方式，提升用户的参与积极性和环保认同感
- **形成闭环**：整合回收 → 再利用 → 公益捐赠等完整链路，让每一份废弃物都能发挥最大价值

### 2.3 路由体系结构

项目采用了分层路由设计，将认证相关页面与业务页面进行分离，并通过路由守卫实现访问控制。完整的路由体系如下：

| 路由路径 | 页面名称 | 访问权限 | 说明 |
|---------|----------|----------|------|
| `/auth` | 登录/注册页 | 公开 | ��立路由，未登录可访问，已登录自动跳转首页 |
| `/` | 首页 | 公开 | 平台主入口，展示服务入口与环保资讯 |
| `/ai-identify` | AI 识别 | 公开 | 上传图片，AI 识别垃圾分类 |
| `/ai-qa` | 环保助手 | 公开 | AI 智能问答，解答环保相关问题 |
| `/science` | 科普页 | 公开 | 环保知识科普内容 |
| `/recycle-booking` | 回收预约 | 公开 | 上门回收预约服务 |
| `/charity` | 公益捐赠 | 公开 | 公益项目捐赠入口 |
| `/upcycle` | 旧物改造 | 公开 | 旧物改造灵感与社区精选 |
| `/orders` | 我的订单 | **需登录** | 订单记录与进度查看 |
| `/profile` | 个人中心 | **需登录** | 用户信息、环保成就、打卡日历 |
| `/faq` | 常见问题 | 公开 | FAQ 问答集锦 |
| `/service-centers/:siteId` | 服务站详情 | 公开 | 各回收服务站点信息 |

路由守卫的核心逻辑包括：已登录用户访问 `/auth` 时自动重定向回首页；未登录用户访问需要认证的页面（如 `/orders`、`/profile`）时，先重定向到登录页并在 URL 中携带原始目标路径，登录成功后可自动跳转回原页面。这种设计既保护了需要登录的功能，又最大程度降低了对用户的干扰。

### 2.4 数据存储与API调用方案

本项目采用了"前端API封装层+本地Node代理服务+本地存储"的混合架构方案：

- **localStorage**：存储用户认证信息（当前用户、可注册用户列表）、个人头像、打卡数据等
- **sessionStorage**：存储AI问答的会话历史等临时数据
- **后端 Node.js 代理服务**：新增的 `backend` 服务充当 API 代理。AI智能问答（对接 DeepSeek）和图像识别（对接 MiniMax）功能通过该代理服务发起请求，从而避免在前端暴露第三方平台的 API Key。
- **前端API封装层**：业务数据接口统一封装在 `src/mock/` 目录下。其中，AI 相关的接口（如 `Aiapi.js` 和 `picAI.js`）已被改造为调用本地后端的接口（`/api/chat` 和 `/api/analyze-image`），其他业务数据暂时仍采用 Mock 数据。

这种架构的优势在于：通过引入轻量级后端，解决了早期纯前端架构下 API Key 暴露的安全隐患，同时后端也能统一提供静态资源服务和 SPA fallback。未来也可以在这个 Node.js 服务上逐步替换掉前端的本地存储，实现完整的数据库对接。

### 2.5 启动与运行

项目采用了前后端分离的运行模式，需要在不同终端分别启动前端与后端：

**后端服务启动：**
```bash
# 进入后端目录
cd backend

# 安装依赖
npm install

# 配置环境变量（复制 .env.example 到 .env 并填入 API Key）
# ...

# 启动服务
node index.js
```
后端服务启动后，默认监听 `http://localhost:8080` 端口。

**前端开发环境启动：**
```bash
# 进入前端项目目录
cd frontend

# 安装项目依赖（如已安装可跳过）
npm install

# 启动开发服务器
npm run dev
```
开发服务器启动后，默认监听 `http://localhost:5173` 端口，可以通过浏览器直接访问。注意：前端的 `mock` API 文件（`Aiapi.js` 等）中 `API_BASE_URL` 可能需要指向后端的 `http://localhost:8080` 才能完成跨域调用（开发环境），如果打包后同源部署，则置空即可。

**生产环境构建与部署：**
```bash
cd frontend
npm run build
```
构建后，后端 `index.js` 会自动托管 `frontend/dist` 目录下的静态资源。直接启动后端即可访问完整的应用。

## 三、核心功能模块详解

### 3.1 首页（HomePage）

首页是整个平台的流量分发中枢，承载了最重要的用户入口功能。页面结构分为以下几个主要区域：

**Hero 区域**：页面顶部的主视觉区域，展示了平台的品牌 slogan“智慧回收，空灵未来”。这里实现了一个独特的打字机特效——品牌描述文案会逐字显示，模拟真实打字效果，放置完成后自动循环播放。这一设计既吸引了用户注意力，又暗示了“AI 技术”的智能化特征。

**核心功能区**：展示了四大核心功能入口卡片，分别是 AI 识别、AI 助手、预约回收、公益捐赠。每个功能卡片都实现了精心设计的 3D 悬浮交互效果——当鼠标移动到卡片上时，卡片会根据鼠标位置产生倾斜的 3D 透视变换效果，创造出沉浸式的交互体验。同时，卡片还配置了发光效果和微动画，进一步强化了科技感与现代感。

**品牌介绍区**：以数据卡片的形式展示了平台的核心竞争力，包括“600+ 城市循环再生项目”“1.5 万吨年处理量”“98% 客户满意度”“AI 识别 99.9% 准确率”等关键指标。

**动态资讯区**：实现了创新的新闻卡片堆叠滚动效果。当用户滚动页面时，多张新闻卡片会产生层叠的视差变换效果，形成一种新颖的信息展示方式。每张卡片都可以点击跳转至对应的功能页面。

### 3.2 AI 识别页（AiIdentifyPage）

AI 识别页是平台的核心功能之一，旨在帮助用户快速判断弃的分类。用户可以通过拍照或上传图片的方式，获取 AI 对废弃物的识别结果和投放建议。

**识别模式切换**：页面提供了“单件识别”和“批量识别”两种模式切换。单件识别适合单个废弃物的精准识别；批量识别功能目前还在开发中，展示了功能的演进规划。

**上传交互**：用户可以通过拍照或相册上传两种方式提交待识别图片。拍照功能调用了系统的相机 API，提供了便捷的移动端拍摄体验。上传区的视觉设计采用了聚焦框元素，引导用户的视觉焦点。

**识别结果展示**：识别结果以结构化的方式展示，包括识别物品名称、置信度（百分比形式）、所属分类（如“可回收物”“有害垃圾”“厨余垃圾”“其他垃圾”）、投放要求、特别提醒等信息。

**结果操作区**：识别结果下方提供了便捷的操作入口，包括“一键预约回收”“公益捐赠”“旧物改造”“环保助手”等，用户可以根据识别结果快速选择下一步操作。

### 3.3 回收预约页（AppointmentPage）

回收预约页是平台的另一核心功能，提供上门回收服务的预约能力。页面采用了多步骤表单的结构设计，逐步引导用户完成预约信息填写。

**步骤一：联系信息**：包括服务地址、联系人姓名、电话号码三个必填字段。地址输入支持详细的门牌号填写，确保回收人员能够准确找到上门位置。

**步骤二：预约时间**：这一步骤实现了较为复杂的时间选择交互。支持月份切换和周切换两种维度的时间选择，通过月历组件展示可选日期。用户可以选择未来 7 天中的可预约日期，其中部分日期会被标记为“已约满”状态，不可选择。时间段提供了多个可预约时段，部分时段也会标记为不可用。

**步骤三：物品详情**：包括物品分类选择（从下拉列表中选择）、预估重量输入（支持自动映射到重量区间）、物品备注与图片上传。重量输入具有智能联动特性——用户输入具体的公斤数后，系统会自动映射到预设的重量区间（如 0-5kg、5-10kg、10-20kg、20kg 以上），并联动显示预计可获得的环保积分。图片上传支持拖拽操作。

**步骤四：信息确认**：在提交前以摘要卡片的形式展示用户填写的所有预约信息，供用户最终确认。

**提交反馈**：提交成功后以弹窗形式展示订单编号、取件码、预计等待时间等信息，并提供跳转到订单页的快捷入口。

### 3.4 个人中心页（ProfilePage）

个人中心页是用户运营的核心阵地，展示了用户的个人信息、环保成就和使用数据。

**头像与基本信息**：用户可以上传自定义头像（本地图片，限制 5MB），系统将头像以 base64 形式存储在 localStorage 中。页面还显示用户的昵称、注册时间等基本信息。

**每日打卡系统**：实现了完整的每日打卡功能。用户点击打卡按钮后，会播放打卡动画，守护天数加 1，并自动记录打卡日期。打卡状态会进行本地持久化，第二天自动解锁新打卡。

**环保成就展示**：以卡片形式展示用户的环保成就数据，包括“守护天数”“总回收次数”“连续打卡记录”等关键指标。

**环境足迹图表**：这一模块实现了动态图表展示。根据用户选择的周期（本周/本月/季度），图表会在柱状图和折线图之间切换，并动态生成模拟的可视化数据。图表支持鼠标悬停交互，悬停时显示具体数值。

**日历视图**：支持月份切换的日历视图，展示每日的回收/打卡记录。用户可以直观地查看历史参与情况。

### 3.5 AI 问答/环保助手页（AiQaPage）

环保助手页为用户提供了一个智能问答的入口，可以解答关于垃圾分类、回收服务、环保知识等方面的疑问。

**多会话支持**：支持创建多个独立的对话会话，每个会话都有独立的历史记录。会话列表默认展示最近 5 条历史记录，支持展开查看更多历史。

**消息交互**：用户可以发送文本消息，AI 会以模拟的“思考中”状态响应，然后返回预置的回复内容。页面还提供了建议问题快捷入口，用户可以一键发送常见问题。

**反馈机制**：每条 AI 回复下方都提供了“有帮助”和“没用”两个反馈按钮，用于收集用户对回答质量的评价。

**侧边栏快捷跳转**：侧边栏提供了跳转到其他功能页的快捷入口，包括 AI 识别、回收预约、FAQ、个人中心、首页等。

### 3.6 订单页（OrdersPage）

## 四、项目亮点与创新特色

### 4.1 前端架构亮点

**完整的单页应用架构**：基于 Vue Router 构建了完整的多路由体系，支持路由守卫、权限控制、懒加载优化。路由守卫实现了基于 `meta` 字段的访问控制，未登录用户访问需要认证的页面时自动重定向并携带原始目标路径，登录后可平滑跳转回原页面。

**统一布局系统**：`ClientLayout` 组件封装了所有 C 端页面的公共布局，包括顶部导航栏、滚动进度条、页面过渡动画、底部页脚等。统一的布局系统保证了用户体验的一致性，也便于后续的维护和迭代。

**旧路径兼容**：通过 `siteMap.js` 定义了旧路径到新路径的映射，如 `/c/home` → `/`、`/c/ai` → `/ai-identify` 等，并配置了兜底重定向逻辑，将未知路径统一引导至首页。这种设计既保护了已发布的链接，又为后续的路径调整提供了灵活性。

**前端API封装层与本地代理**：业务数据接口统一封装在 `src/mock/` 目录下。最核心的AI相关接口（如图片识别、流式智能问答）已被迁移到本地的 Node.js Express 后端进行代理转发，彻底解决了纯前端直接调用第三方AI服务的 API Key 泄漏风险。其他诸如用户信息等仍以封装的形式保留，便于后续完全迁移到后端数据库。这种渐进式后端迁移的设计，使得前端开发可以保持灵活性，又增强了系统的安全性。

### 4.2 交互设计亮点

**动态导航栏**：导航栏实现了一个会移动的高亮滑块。当路由发生变化或鼠标 hover 到不同的导航项时，高亮滑块会平滑地移动到对应的位置。这种设计比传统的静态高亮更具动感和现代感，也更好地引导了用户的视觉焦点。

**打字机特效**：首页 Hero 区域的品牌描述文案实现了逐字显示的打字机效果。通过 `setTimeout` 控制字符的逐个显示，放置完成后自动循环播放。打字速度经过了精心调校（58ms/字符），在可读性和动画感之间取得了良好的平衡。代码还考虑了 `prefers-reduced-motion` 无障碍偏好，对于减少动画偏好的用户会直接显示完整文案。

**3D 卡片悬浮**：核心功能区的四张卡片实现了 3D 悬浮效果。当鼠标移动到卡片上时，根据鼠标位置计算机卡片相对于中心的偏移量，应用 `perspective(1000px) rotateX() rotateY()` 的 3D 变换。这种效果比传统的简单放大更具空间感和沉浸感。

**新闻卡片堆叠滚动**：动态资讯区实现了创新的堆叠滚动效果。当容器滚动时，每张卡片会根据滚动位置产生缩放和位移的变换，形成层叠的视差效果。代码使用了 `requestAnimationFrame` 进行性能优化，并配置了 `will-change` 属性提示浏览器进行渲染优化。

**滚动揭示动画**：项目封装了 `useRevealOnScroll` 组合式函数，为页面元素提供“进入视口后淡入显现”的动画效果。页面元素只需标记 `data-reveal` 属性，即可自动获得滚动揭示动画。揭示动画支持自定义延迟时间（如 `style="--reveal-delay: 120ms"`），便于错开不同元素的动画时机。

### 4.3 表单交互亮点

**日期时间选择器**：回收预约页的日期时间选择器实现了较为复杂的交互逻辑。支持月份和周两种维度的切换，通过月历组件展示可选日期。日期和时段都有“约满”的禁用状态处理，用户不可选择已约满的时间。

**重量区间联动**：用户在重量输入框中输入具体数值后，系统会自动计算并映射到预设的重量区间（如 0-5kg、5-10kg、10-20kg、20kg 以上），同时联动显示预计可获得的环保积分。这种智能联动设计提升了表单的填写效率。

**表单校验**：封装了 `appointmentValidation.js` 工具函数，对预约表单的各个字段进行校验。校验消息会在用户提交时以弹窗形式展示，引导用户补全信息。

### 4.4 用户运营亮点

**登录引导弹窗**：对于未登录用户，在进入站点约 3 秒后会弹出登录引导弹窗。弹窗提供了“先去逛逛”（关闭弹窗）和“去登录”两个选择。这种设计既不会在用户刚进入时强制打断浏览，又在合适的时机主动引导登录。

**每日打卡系统**：个人中心实现了完整的每日打卡系统。用户点击打卡后会有专门的动画效果，守护天数和连续打卡记录会自动更新，并自动滚动到日历区域高亮今天的记录。打卡数据会持久化到 localStorage，用户第二天再次访问时可以继续打卡。

**环保成就系统**：个人中心展示了用户的环保成就数据，包括守护天数、总回收次数、最长连续记录等。这些数据以卡片形式展示，配合可视化图表，直观地呈现用户的环保贡献。

### 4.5 多端适配亮点

**响应式设计**：项目充分考虑了桌面端和平板/移动端的显示差异。核心功能区的四张卡片在桌面端以 2x2 网格排列，在移动端会自动切换为单列布局。月历选择器也有对应的响应式适配。

**移动端导航**：导航栏为移动端提供了独立的抽屉菜单，点击汉堡图标可以展开侧边抽屉。抽屉包含主导航跳转、账户菜单、退出登录等入口，支持点击遮罩关闭或按 Escape 键关闭。

## 五、关键技术实现解析

### 5.1 Vue 3 Composition API 的应用

项目全面采用了 Vue 3 的 Composition API 进行开发，具体体现在 `<script setup>` 语法糖的广泛使用。`<script setup>` 是 Vue 3.2+ 引入的语法糖，可以将组件的逻辑更简洁地组织在同一个代码块中，同时获得更好的类型推导支持。

一个典型的例子是回收预约页的状态管理：

```javascript
const form = reactive({
  category: "",
  weight: "",
  date: "",
  period: "",
  address: "",
  contactName: "",
  phone: "",
  note: "",
});
```

这种响应式对象使得表单数据具有自动追踪能力，无需手动调用 `setState` 或者编写繁复的 watch 处理。同时，页面的多个计算属性都是基于这个响应式对象构建的：

```javascript
const selectedDateText = computed(() => {
  if (!selectedDateCard.value) return "待选择";
  return `${selectedDateCard.value.monthText}${selectedDateCard.value.dayNumber}日（${selectedDateCard.value.weekday}）`;
});
```

### 5.2 自定义组合式函数

项目封装了多个自定义组合式函数（Custom Composables），用于逻辑复用和代码组织。

**useRevealOnScroll**：这是最核心的组合式函数之一，提供了“元素进入视口后淡入显现”的动画效果。其实现原理是：

1. 监听页面的滚动事件（或使用 Intersection Observer API）
2. 检测目标元素是否进入视口
3. 为进入视口的元素添加 `revealed` 类名
4. CSS 中定义 `.revealed` 类名的动画（如 opacity 从 0 到 1、transform 从 translateY(20px) 到 0）

页面中使用方式是：

```javascript
const pageRef = ref(null);
useRevealOnScroll(pageRef);
```

```html
<section ref="pageRef" data-reveal>...</section>
```

这种设计使得页面可以零碎地添加滚动揭示效果，而不需要为每个元素单独编写观察逻辑。

### 5.3 本地认证系统

项目的认证功能完全基于浏览器本地存储实现，不需要真实的后端服务。认证数据存储在 localStorage 中，包含两个 key：

- `szt_user`：当前登录用户的信息（JSON 格式）
- `szt_users`：所有已注册用户的信息列表（JSON 格式）

项目启动时会自动初始化一个演示账号：

- 账号：`user@szt.com`
- 密码：`123456`

登录时的核心逻辑是验证账号密码是否匹配列表中的用户记录，匹配成功后将用户信息写入 `szt_user` 并更新 UI 状态。登出时只需要清除 `szt_user` 的值。

这种本地认证方案的优势是开发阶段可以完全模拟完整的登录流程，劣势是安全性较低、不支持多设备同步。后续对接真实后端时，只需要将 localStorage 操作替换为 API 调用即可。

### 5.4 前端API封装层设计

项目的前端 API 封装层采用了分层设计，不同功能的数据接口分别封装在 `src/mock/` 的不同文件中：

| 文件 | 功能 |
|------|------|
| `clientApi.js` | 核心业务数据（首页内容、预约元数据、订单列表、用户资料等，目前由本地存储提供 Mock 数据） |
| `Aiapi.js` | AI 能力相关（向本地 Node 后端 `/api/chat` 发起请求，获取流式回复） |
| `picAI.js` | 图像识别（向本地 Node 后端 `/api/analyze-image` 提交 base64 数据） |
| `timeApi.js` | 时间相关（日期生成、日历数据等） |

这种分层设计保证了后续可以平滑过渡到完整的后端架构。接口大多设计为返回 Promise，保持与真实网络请求相同的异步风格。例如本地数据接口的模拟：

```javascript
export async function fetchHomeData() {
  // 模拟 API 延迟
  await new Promise(resolve => setTimeout(resolve, 400));
  return HOME_DATA;
}
```

而对于 AI 能力，则直接通过 `fetch` 调用后端的 Express 代理：
```javascript
export function streamChat(userMessage, history, onChunk, onComplete, onError) {
  // 构建消息...
  fetch(`${API_BASE_URL}/api/chat`, { /* ... */ })
    // 解析流式响应
}
```

## 六、Web 网页使用情况概述

### 6.1 应用架构

本项目为单页应用（Single Page Application，SPA），所有功能均在浏览器内通过前端路由完成切换，无需整页刷新。与传统多页面应用相比，SPA 具有以下优势：

- **用户体验流畅**：页面切换无刷新感，配合过渡动画可以实现原生应用级的流畅度
- **前后端分离**：前端通过API封装层独立实现，可以通过本地API直接演示
- **路由灵活**：前端路由可以精细控制页面状态，支持深链接

### 6.2 核心业务流程

用户使用平台的典型流程包括以下几条主线：

**主线一：浏览 → 登录 → 账户体系**

1. 用户进入首页或其他公开页面
2. 未登录状态下，浏览约 3 秒后弹出登录引导弹窗
3. 用户选择“去登录”跳转到 `/auth` 页面
4. 输入演示账号（`user@szt.com` / `123456`）完成登录
5. 登录成功后自动跳转回原页面或首页
6. 用户可以访问订单页、个人中心等需要登录的功能

**主线二：识别 → 分类 → 预约**

1. 用户进入 AI 识别页
2. 上传待识别的废弃物图片
3. 系统返回识别结果（分类、投放建议等）
4. 用户点击“一键预约回收”
5. 跳转到回收预约页，预填部分信息
6. 用户补充完整预约信息并提交

**主线三：内容教育 → 功能转化**

1. 用户浏览 FAQ、科普页、旧物改造等内容页
2. 被内容激发环保意识
3. 回流到核心功能页（AI 识别、回收预约、公益捐赠等）

**主线四：用户运营闭环**

1. 用户登录后进入个人中心
2. 每日打卡获取守护天数
3. 查看环保数据图表
4. 查看历史订单记录
5. 形成持续使用习惯

### 6.3 响应式适配

项目针对不同设备屏幕尺寸进行了响应式适配，主要包括：

- **桌面端（>1180px）**：核心功能区卡片 2x2 布局，月历选择器 7 列
- **平板端（720px-1180px）**：核心功能区卡片单列布局，统计数据卡片 2 列
- **移动端（<720px）**：布局完全重排，输入表单全宽度，日期选择器 4 列或 2 列

「收智通」是一个功能完整、采用前后端分离架构的 C 端环保回收平台，具备以下特点：

- **清晰的架构**：基于 Vue 3 + Vue Router 构建前端，配合 Node.js Express 提供本地后端代理与静态服务托管。
- **丰富的功能**：覆盖 AI 识别、预约回收、个人中心、订单管理、公益捐赠等完整的业务功能。
