# 收智通前端项目逻辑说明

## 1. 项目整体概况

这个项目是一个基于 **Vue 3 + Vue Router + Vite** 的单页前端项目，整体定位为 **“收智通” C 端环保回收平台**。

当前项目已经具备：

- 完整的页面路由体系
- 登录 / 注册 / 退出逻辑
- 部分登录鉴权能力
- 多个核心业务页面的前端交互原型
- 大量基于 mock 数据和 localStorage 的演示流转

从代码结构上看，它更接近一个 **高保真前端业务原型**，而不是完全对接后端的正式业务系统。

---

## 2. 技术栈与入口逻辑

### 2.1 技术栈

从 `package.json` 可以看出项目使用了：

- `vue`
- `vue-router`
- `vite`

### 2.2 应用启动流程

入口文件在 `src/main.js`。

启动时主要做了以下几件事：

1. 引入 `App.vue`
2. 引入路由 `router`
3. 执行 `initAuthSeed()` 初始化演示账号
4. 挂载 Vue 应用到页面

也就是说，项目一启动，就会先往 `localStorage` 中注入一个默认演示账号，方便体验登录流程。

## 3. 全局页面承载逻辑

### 3.1 根组件职责

根组件在 `src/App.vue`。

它有两个核心职责：

#### 1）承载整个路由系统

通过 `<RouterView />` 来渲染当前路由对应的页面。

#### 2）处理全局登录引导弹窗

当用户满足以下条件时，会在进入站点约 3 秒后自动弹出登录引导框：

- 当前没有登录
- 当前页面不是 `/auth`
- 当前浏览器会话里没有弹过这个提示

### 3.2 登录弹窗的行为

弹窗组件是 `src/components/common/LoginPromptModal.vue`。

用户有两个选择：

- **先逛逛**：关闭弹窗，不跳转
- **去登录 / 注册**：跳转到 `/auth?redirect=当前页面完整路径`

这个设计的目的是：

- 不一进页面就强制打断用户
- 但在浏览几秒后主动引导其进入登录体系
- 登录后仍然可以回到原本浏览的页面

## 4. 路由结构与跳转逻辑

路由定义在 `src/router/index.js`。

### 4.1 路由整体结构

项目路由分为两大类：

#### A. 独立认证页

- `/auth`：登录 / 注册页

#### B. C 端主站页面

这些页面统一挂在 `ClientLayout` 下：

- `/`：首页
- `/ai-identify`：AI识别
- `/ai-qa`：环保助手 / AI问答
- `/recycle-booking`：回收预约
- `/charity`：公益捐赠
- `/upcycle`：旧物改造
- `/science`：科普页
- `/orders`：我的订单
- `/profile`：个人中心
- `/faq`：常见问题
- `/service-centers/:siteId`：服务站详情

### 4.2 统一布局逻辑

`src/layouts/ClientLayout.vue` 负责所有 C 端页面的公共布局，包含：

- 顶部导航栏 `NavBar`
- 顶部滚动进度条
- 页面切换区
- 底部页脚 `SiteFooter`

同时页面切换使用了带淡入淡出效果的过渡动画。

### 4.3 滚动行为

路由层定义了：

- 每次切页自动滚动到顶部
- 使用 `smooth` 平滑滚动

因此从用户体验上看，页面切换是比较顺滑的。

## 5. 登录鉴权与守卫逻辑

### 5.1 当前用户获取方式

项目通过 `src/utils/auth.js` 中的 `getCurrentUser()` 从 `localStorage` 中读取当前登录用户。

### 5.2 守卫逻辑

路由守卫定义在 `src/router/index.js` 的 `beforeEach` 中。

#### 规则一：已登录用户不能再访问 `/auth`

如果用户已经登录，再访问登录页，会被直接重定向回首页。

#### 规则二：部分页面需要登录

当前加了 `requiresAuth` 的页面有：

- `/orders`
- `/profile`

未登录访问这些页面时，会被自动跳转到：

- `/auth?redirect=原目标页面`

这样登录后就能重新跳回原页面。

### 5.3 退出登录逻辑

退出登录入口在导航栏账户菜单里。

触发后会：

1. 清除当前登录用户
2. 同步导航显示状态
3. 如果当前不在首页，则跳回首页

## 8. 登录 / 注册逻辑

认证逻辑主要在：

- `src/views/auth/AuthPage.vue`
- `src/utils/auth.js`

### 8.1 数据存储方式

当前没有后端，所有认证数据都存在 `localStorage`：

- `szt_user`：当前登录用户
- `szt_users`：所有可登录用户

### 8.2 默认演示账号

项目初始化时会写入演示账号：

- 账号：`user@szt.com`
- 密码：`123456`

### 8.3 登录流程

登录时会做以下校验：

- 账号至少 4 位
- 密码不能为空

若校验通过，则调用 `login()`：

- 如果账号密码匹配，写入当前用户信息
- 然后跳到 `redirect` 指向的页面
- 如果没有 `redirect`，就回首页

### 8.4 注册流程

注册时会校验：

- 昵称不能为空
- 账号至少 4 位
- 密码至少 6 位
- 两次密码输入一致
- 账号不能重复

注册成功后不会自动登录，而是：

1. 显示“注册成功，请登录”
2. 自动切回登录模式
3. 自动把刚注册的账号密码填入登录表单

这是一种“注册后再确认登录”的流程。

## 10. 回收预约页逻辑

页面在 `src/views/client/AppointmentPage.vue`。

这是一个较完整的表单式业务页面。

### 10.1 初始化逻辑

页面加载时会调用 `fetchAppointmentMeta()`，获取：

- 回收品类
- 重量区间
- 时段信息
- 提示信息

并自动设置默认值。

### 10.2 用户交互内容

#### 1）联系信息输入

包括：

- 地址
- 联系人姓名
- 电话号码

#### 2）日期选择

用户可以选择未来 7 天中的可预约日期。

其中后两天会被标记为“约满”，不能选中。

#### 3）时间段选择

页面提供多个可预约时段，禁用项不可点击。

#### 4）重量输入联动

用户输入大致重量后，会自动映射到区间：

- 0-5kg
- 5-10kg
- 10-20kg
- 20kg以上

并联动预计环保积分。

#### 5）文件选择

支持本地选择图片文件，但当前没有上传到后端，只做了本地名称展示。

### 10.3 提交流程

用户提交后：

1. 先调用表单校验函数进行校验
2. 如果有问题，弹出校验提示框
3. 如果通过，则调用 `submitAppointment()`
4. 返回 mock 结果，例如：
   - 订单号
   - 取件码
   - 预计积分
   - 预计上门时间

### 10.4 当前限制

当前预约提交只是前端 mock 成功，不会真的写入订单页的数据源。

也就是说它形成了“提交反馈闭环”，但还没有形成“订单同步闭环”。

## 12. 个人中心页逻辑

页面在 `src/views/client/ProfilePage.vue`。

这是一个偏“用户运营”和“个人成就展示”的页面。

### 12.1 数据来源

数据来源分为两类：

#### 来自 mock 的数据

通过 `fetchProfileData()` 获取一些基础展示信息。

#### 来自 localStorage 的数据

包括：

- `guardianDays`：守护天数
- `userAvatar`：用户头像
- `lastCheckInDate`：最近打卡日期

### 12.2 主要交互

#### 1）头像上传

用户可选择本地图片作为头像，系统会：

- 校验是否为图片
- 校验大小是否超过 5MB
- 读成 base64
- 保存到 localStorage

#### 2）每日打卡

用户点击打卡按钮后：

- 若今天已打卡，则弹提示
- 若未打卡，则：
  - 播放动画
  - 守护天数加一
  - 保存今日日期
  - 平滑滚动到日历区域并高亮今天

#### 3）周期切换

环境足迹图表支持按：

- 本周
- 本月
- 季度

切换不同维度的数据表现。

#### 4）月份切换

用户可以切换月份，动态生成月历数据。

### 12.3 当前限制

Profile 页很多数据属于本地状态或随机展示数据，不是稳定的后端业务数据。

#### 2）个人中心

#### 3）AI识别

## 21. 整个项目的主线业务逻辑

从用户视角，这个项目大致有四条主要主线。

### 主线一：浏览 → 被引导登录 → 进入账户体系

1. 用户进入首页或公开页面
2. 未登录时经过约 3 秒触发登录引导
3. 跳转到 `/auth`
4. 登录成功后跳回原页面或首页
5. 用户可以继续进入订单页、个人页等账户功能

### 主线二：识别 → 分类理解 → 发起预约

1. 用户进入 AI识别页
2. 上传图片识别物品类别
3. 得到分类建议和处理说明
4. 跳去回收预约页
5. 发起上门回收申请

### 主线三：内容教育 → 功能转化

1. 用户浏览 FAQ、科普、旧物改造等内容页
2. 被教育或激发环保行为
3. 回流到实际功能页，如：
   - AI识别
   - 回收预约
   - 公益捐赠

### 主线四：登录后用户运营

1. 用户登录
2. 进入个人中心
3. 打卡、查看环保数据
4. 查看订单记录
5. 建立持续使用习惯

## 22. 当前代码中值得注意的逻辑问题

在梳理过程中，可以看到几个尚未完全接好的点。

### 22.1 订单页筛选逻辑未完全接入

虽然页面有：

- tab 切换
- 服务类型筛选
- 时间范围筛选

但目前真正参与过滤的只有关键词搜索。

### 22.2 回收预约成功后未进入订单数据

预约页提交成功只展示结果，并没有把新订单同步追加到订单列表。

### 22.3 FAQ 浮动入口文案与目标页不完全一致

FAQ 页悬浮按钮文案偏向“问答助手”，但跳转目标更像 AI识别入口，语义上略有不一致。

### 22.4 AI问答页未使用已有 mock AI API

mock 数据层已经存在：

- `fetchAiQuickQuestions()`
- `askAiAssistant()`

但当前聊天页没有接这些函数，而是使用本地随机回复数组。

### 22.5 Profile 页数据稳定性较弱

部分图表和日历数据依赖随机生成，因此更偏展示动画，不适合作为真实用户历史数据。

## 23. 最终总结

这个项目本质上是一个 **环保回收平台 C 端单页应用**，目前已经搭建出了比较清晰的产品结构与交互骨架。

它已经具备：

- 完整的前端路由体系
- 比较成熟的登录鉴权流程
- 公共导航和布局系统
- 多个核心业务页面的高保真交互
- 一定程度上的本地状态闭环

但从业务完整度看，除了登录注册和路由鉴权外，很多功能仍然停留在：

- mock 数据驱动
- localStorage 驱动
- 前端演示流

所以它当前最适合的定位是：

> 一个已经具备较强产品表达力和交互表现力的 C 端环保平台前端原型项目。

如果后续要继续推进，可以优先补齐：

1. 预约成功 → 订单同步
2. 订单页 tab / 筛选真实生效
3. AI问答接 mock API 或真实接口
4. 公益捐赠提交流转闭环
5. 用户中心数据稳定化
